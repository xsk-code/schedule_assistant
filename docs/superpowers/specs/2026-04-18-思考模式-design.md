# 思考模式：智能多轮任务澄清功能

**版本**：v1.0
**日期**：2026-04-18
**作者**：四化节奏师产品设计

---

## 1. 概述

### 1.1 背景与目的

当前用户在使用任务拆分功能时，输入的任务描述往往较为简单，导致 AI 生成的任务拆解方案缺乏深度和针对性。

为了提升方案质量，引入「思考模式」—— 一个多轮对话式的任务澄清流程。AI 通过主动追问，引导用户补充关键信息，从而生成更精准、更实用的任务拆分方案。

### 1.2 核心价值

- **深度理解**：AI 通过追问挖掘用户真实需求
- **用户主导**：用户掌控对话节奏，可随时主动结束
- **效果导向**：聚焦任务场景和期望效果，而非泛泛而谈

## 2. 产品设计

### 2.1 模式切换

在 TaskInput 组件右下角添加模式切换开关：

```
┌─────────────────────────────────────┐
│                                     │
│         [任务输入框]                 │
│                                     │
│                      [⚡简单] ↔ [💭思考] │
└─────────────────────────────────────┘
```

- **默认状态**：简单模式（保持现有体验）
- **切换动作**：点击即可切换，无需保存
- **视觉反馈**：切换按钮高亮当前模式

### 2.2 简单模式

保持现有单次输入 → 单次分析 → 结果展示的流程不变。

### 2.3 思考模式交互流程

```
用户输入任务
    ↓
AI 分析任务，判断需要澄清的问题
    ↓
展示问题 + 选项卡片 + "其他"输入框
    ↓
用户选择选项 / 输入其他内容 / 跳过 / 主动结束
    ↓
AI 继续追问或判断信息足够
    ↓
达到结束条件 → 生成完整任务拆分方案
    ↓
进入结果展示流程
```

**布局说明：**
- 原始任务以卡片形式固定在顶部，作为对话锚点
- 每轮问题独占一个卡片区域
- 选项采用**垂直堆叠**、**每行一个**的布局
- 前三个为预设选项，第四个为"其他"（带输入框）
- "跳过"和"确认"按钮在同一行，"确认"按钮默认禁用

### 2.5 结束条件

AI 自行判断以下任一条件满足时结束对话：

1. **信息充分**：AI 判断当前收集的信息足以生成高质量方案
2. **轮次上限**：达到 5 轮强制结束
3. **用户主动**：用户点击"信息足够，生成方案"

### 2.6 选项卡片交互规则

- **单选**：每次只能选择一个选项
- **选中态**：卡片边框高亮 + 背景色变化
- **其他输入**：当选中"其他"时，展开输入框
- **跳过**：当问题不适用时，用户可跳过
- **继续确认**：用户回答后，需点击"确认"按钮发送

### 3.2 核心类型定义

```typescript
// 对话上下文
interface ConversationContext {
  originalTask: string;       // 用户原始任务
  collectedInfo: CollectedItem[];  // 已收集的信息
  currentRound: number;       // 当前轮次
  isComplete: boolean;        // 对话是否结束
  isGenerating: boolean;      // 是否正在生成方案
}

interface CollectedItem {
  question: string;           // AI 提问
  answer: string;             // 用户回答
  timestamp: number;
}

// AI 追问响应
interface AIQuestion {
  question: string;           // 当前问题
  options: string[];          // 选项列表（3-4个）
  hint: string;               // 辅助说明（可选）
}

// 最终生成请求
interface FinalGenerationRequest {
  originalTask: string;
  collectedInfo: CollectedItem[];
  sihuaInfo: SihuaInfo;
}
```

### 3.4 ConversationService Prompt 设计

**系统角色**：
```
你是一个任务需求澄清助手。你的职责是通过精准的追问，帮助用户明确任务的关键信息。

【追问原则】
1. 每次只问 1 个问题
2. 问题要具体、可回答
3. 选项要覆盖常见情况，提供"其他"选项
4. 当收集到足够信息时，主动结束对话

【判断信息充分的标准】
- 任务目标已明确
- 关键约束条件已确认（如时间、资源）
- 期望效果已了解

【输出格式】
当需要继续追问时，输出：
{
  "needsMoreInfo": true,
  "question": "问题内容",
  "options": ["选项1", "选项2", "选项3", "其他"],
  "reasoning": "为什么问这个问题"
}

当信息足够时，输出：
{
  "needsMoreInfo": false,
  "reasoning": "为什么信息已经足够",
  "summary": "你对任务的理解总结"
}
```

### 3.5 API 调用策略

**Token 优化**：
- 每轮只发送：`当前问题回答` + `原始任务` + `已收集信息`（结构化小对象）
- 不发送对话历史，最大化节省 token

**请求示例**（第 N 轮）：
```json
{
  "task": "准备期末考试",
  "collectedInfo": [
    {"question": "是哪门课程？", "answer": "高等数学"},
    {"question": "距离考试还有多久？", "answer": "一周内"}
  ],
  "maxRounds": 5
}
```

### 3.6 与现有流程的集成

```
用户点击"开始分析"
    ↓
检查模式
    ↓
简单模式 → 调用现有 analyzeTask()
思考模式 → 调用 useConversation.startConversation()
    ↓
两者最终都 → 展示 AnalysisResult
```

## 5. 错误处理

| 场景 | 处理方式 |
|------|----------|
| API 超时 | 重试 1 次，失败后显示错误提示，提供"重试"按钮 |
| API 返回格式错误 | 记录日志，提示用户稍后重试 |
| 网络断开 | 提示网络问题，保存当前对话状态 |
| 用户刷新页面 | 对话状态丢失，需要重新开始 |
